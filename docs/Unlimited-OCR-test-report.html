📄 Unlimited-OCR 本地化測試報告

日期：2026-06-27 ｜ 環境：WSL2 Ubuntu-24.04 ／ RTX 5070 Ti（Blackwell sm_120）／ SGLang triton backend ｜ 規劃文件：docs/superpowers/specs/2026-06-27-ocr-testing-plan-design.md

1. 實驗目的

在正式容器化部署（H100）前，系統化驗證本地 Unlimited-OCR（SGLang 路線）的 辨識品質 與 服務穩定性，找出模型可信範圍與弱點。刻意 不採用公開 benchmark（疑訓練汙染），改以難度分層的自建語料評估。

2. 規劃

  • 品質（難度分層）：simple（有乾淨文字層）→ 以 PyMuPDF 抽文字層為 pseudo-GT，自動算 CER／覆蓋率／順序不敏感相似度（CJK 以單字元 tokenize，避免空白切詞失準）；hard（表格/公式/圖/掃描）→ 人工 rubric 評分（6 項各 0–5）。
  • 穩定性：S1 長跑不退化/不洩漏（+ tok/s 與 GPU mem 速度曲線）、S2 迴圈頁防護、S3 不 crash、S4 錯誤回復。
  • 語料：7 類別共 9 份（學術、CJK、表格、掃描、專利、10-K、human 合約/手寫）；公開可抓的自動下載，難找的使用者自備。

3. 流程

  • 雙 venv（transformers / sglang）以 uv 建置；SGLang 在 Blackwell 採 triton backend + CUDA Toolkit 12.8 runtime JIT。
  • 每頁經 SGLang 串流推論（max_tokens=4096、逾時 30s 後備），存純文字／原始輸出／疊框圖。
  • 品質軌：python -m bench.quality；穩定軌：python -m bench.stability --longrun-pages 200。
  • 過程中修正：CJK-aware 指標、S2 改量記憶體增量門檻。

5. 結論

  • 乾淨印刷體/born-digital 文件辨識極佳：英文 CER 1.3%、中文 14%；表格結構、公式、閱讀順序普遍 5/5。適合作為此類文件的 OCR 服務。
  • 服務穩定可靠：200 頁長跑不退化、GPU 記憶體不洩漏；迴圈頁由 max_tokens 乾淨截斷；對異常輸入不 crash、中止後可回復。
  • 已知弱點（人工發現）：
    • 掃描雜訊／印章會干擾文字（廠商/地名/電話辨識錯）。
    • 密集表格頁出現重複生成（重複判 title、大段文字重複），會觸發 max_tokens 上限 → 該頁結果可能不完整（防線有效擋住，但需監看 capped 標記）。
    • 空白區域偶被判為 text；bounding box 完整度（chart/印章/figure title）有改善空間。
  • 建議：部署可優先用於印刷體/表格類文件；對低品質掃描/印章文件建議前處理或人工複核；服務面監看 capped/逾時頁標記。
